# 侧边栏日历组件设计

## 背景

当前博客使用 `Hexo + argon` 主题，侧边栏已经承载公告、站点概览等卡片模块，但还没有一个原生的日历类组件。

用户希望新增一个可复用的日历组件，首先接入侧边栏，满足以下核心需求：

- 以月历表的形式展示日期，而不是只显示“今天”
- 明显高亮今日
- 点击任意日期后，切换下方详情
- 详情展示标准黄历信息：
  - 公历
  - 农历
  - 星期
  - 节气
  - 节日
  - 宜
  - 忌
- 手机端也显示完整月历
- 组件本身要可复用，方便其他模块直接引入，而不是把逻辑写死在侧边栏模板中

## 目标

- 为 `argon` 主题新增一个可复用的日历组件。
- 组件第一落点为侧边栏卡片，但设计上允许后续被首页、独立页面或其他模块复用。
- 使用本地计算型能力完成农历、节日、节气、宜忌展示，避免把核心体验绑死在第三方 iframe 或远端接口上。
- 保持和现有侧边栏卡片一致的视觉语言，不做割裂的外挂式组件。

## 非目标

- 第一版不接入远端黄历接口。
- 第一版不做节假日排班查询、天气联动、提醒功能。
- 第一版不扩展为全站通用页面壳子改造。
- 第一版不强求支持多实例复杂联动场景，例如一个页面同时挂多个彼此同步的月历。

## 数据与技术选择

本次组件底层使用 `lunar-javascript`。

选择原因：

- 该库能力与需求高度匹配，覆盖农历、节日、节气、每日宜忌等信息。
- 适合静态博客场景，不依赖服务端运行时。
- 比直接嵌入第三方黄历页面更可控，样式和交互也更容易融入当前主题。

不采用 `tyme4ts` 作为第一方案的原因：

- 当前需求重点在“黄历展示”而不是更泛化的日历体系抽象。
- 用户明确要求宜忌，`lunar-javascript` 对该场景更直接。

## 现状摘要

- 侧边栏模板位于 `themes/argon/layout/sidebar.ejs`。
- 主题已经存在 `layout/_partial/` 结构，适合新增可复用 partial。
- 现有侧边栏模块主要以 `card` 形式组织，视觉和结构入口明确。
- 项目当前不是完整 git 工作树，`D:/blog/.git` 缺失，仅存在 `D:/blog/.deploy_git`，因此设计文档可以落盘，但无法按常规流程提交到当前仓库。

## 设计总览

组件采用“三层拆分 + 一处接入”的方式：

1. 可复用模板 partial
2. 独立组件脚本
3. 独立组件样式
4. 侧边栏作为第一处使用方接入

这样可以把“组件本体”和“组件放在哪里展示”解耦，避免后续复用时重复复制模板和脚本。

## 组件结构

组件分为四个区域：

### 1. 顶部栏

- 左侧显示当前年月，例如 `2026 年 6 月`
- 右侧显示上个月 / 下个月切换按钮
- 初始进入时默认显示当前月份

### 2. 月历表

- 采用 7 列布局，星期顺序为 `一二三四五六日`
- 每个日期格包含两层信息：
  - 主信息：公历日期数字
  - 次信息：农历日，若存在节日或节气则优先显示节日/节气名称
- 今日需要有明确高亮
- 当前选中日期需要有独立选中态，并与“今天”态区分

### 3. 详情区

点击某个日期后，下方详情区显示该日的标准黄历信息：

- 公历完整日期
- 星期
- 农历日期
- 节气
- 节日
- 宜
- 忌

若某项为空，例如当天无节气，则不强制单独占一行，以控制卡片高度。

### 4. 移动端表现

- 手机端继续显示完整月历
- 月历格尺寸缩小，但仍保持可点击
- 详情区继续作为卡片下半部分展示
- 不改为弹窗，不改为折叠面板

## 交互规则

### 初始状态

- 默认进入当前月份
- 默认选中今天
- 详情区展示今天的黄历信息

### 点击日期

- 点击任意日期后，该日期进入选中态
- 详情区立即切换到该日数据
- 若点击的是今天，则今天态和选中态可以叠加，但需要通过视觉层级区分
- 若点击的不是今天，则今天仍保留弱化的“今日标记”

### 切换月份

- 点击上个月 / 下个月按钮后，切换月历表
- 切月后默认选中该月 `1 号`
- 详情区同步切换到对应日期信息
- 若回到当前月份，且用户尚未再次主动选择别的日期，可以恢复默认选中今天

### 日期格内容优先级

- 若当天存在节日，优先显示节日
- 否则若存在节气，显示节气
- 否则显示农历日

### 非本月日期

第一版不补前后月灰色日期，只显示当前月份日期。

这样可以获得以下收益：

- 侧边栏空间更稳定
- 手机端布局更易控
- 逻辑更简单，第一版交互成本更低

## 组件化设计

### 模板层

新增一个可复用 partial，例如：

- `themes/argon/layout/_partial/calendar.ejs`

该 partial 只负责输出组件骨架，不承载复杂计算逻辑。

### 脚本层

新增独立组件脚本，负责：

- 初始化组件
- 渲染某个月份的日期网格
- 高亮今天
- 响应点击日期和切月操作
- 调用 `lunar-javascript` 生成当天详情数据
- 更新详情区内容

脚本不应写成“只能服务侧边栏唯一实例”的结构，而应尽量暴露成基于容器挂载的初始化方式，方便后续复用。

### 样式层

新增独立组件样式文件，并统一使用前缀，例如：

- `.lq-calendar-*`

这样可以避免污染现有 `argon` 样式命名，也便于日后移动或复用。

### 配置层

第一版组件保留少量对外配置即可，例如：

- `showDetail`
- `showNavigator`
- `initialDate`
- `mobileFull`

这足以覆盖侧边栏、首页卡片、独立页面等常见使用场景，同时避免过早设计过多接口。

## 侧边栏接入方式

侧边栏只作为组件的使用方，不承载组件本体逻辑。

建议做法：

- 在 `themes/argon/layout/sidebar.ejs` 中新增一张侧边栏 `card`
- 在这张 `card` 内部通过 `partial('_partial/calendar')` 引入组件模板

这样处理后：

- 侧边栏结构仍清晰
- 组件可以单独维护
- 未来其他页面复用时无需复制大段模板代码

## 文件落位建议

建议改动点如下：

- `themes/argon/layout/_partial/calendar.ejs`
  - 日历组件模板骨架
- `themes/argon/layout/sidebar.ejs`
  - 新增日历卡片接入点
- 主题静态资源中的独立脚本文件
  - 日历组件交互与 `lunar-javascript` 数据读取
- 主题静态资源中的独立样式文件
  - 日历组件样式和响应式规则

具体文件名实现阶段可根据主题现有资源组织方式微调，但应保持“模板、样式、脚本分离”的结构。

## 实施顺序

建议按以下顺序实现：

1. 新增组件模板骨架并在侧边栏完成接入
2. 新增组件样式，先完成静态布局和响应式结构
3. 接入 `lunar-javascript`，实现当前月渲染、今日高亮和日期详情切换
4. 增加切月逻辑
5. 收尾处理长节日名、移动端细节和视觉打磨

这样可以优先跑通结构，再逐步补齐数据和交互，风险最低。

## 错误处理与边界

- 若某一天不存在节气或节日，不显示空白标题行
- 若 `lunar-javascript` 数据异常，组件至少仍能显示公历日期和基础结构，避免整张卡片空白
- 若脚本初始化失败，不应影响侧边栏其他模块渲染
- 若移动端空间过窄，优先保证日期点击和详情可读，不强行保留过多装饰

## 验证标准

完成实现后应满足以下标准：

1. 侧边栏出现一张新的日历卡片，风格与现有 `argon` 卡片兼容
2. 默认显示当前月份，并高亮今天
3. 默认详情区展示今天的公历、农历、星期、节气、节日、宜、忌
4. 点击任意日期后，详情区即时切换
5. 点击上个月 / 下个月后，月历表和详情区同步更新
6. 手机端仍显示完整月历，不出现明显挤压或破版
7. 组件逻辑不写死在 `sidebar.ejs`，而是可以被其他模板复用

## 风险与取舍

- 第一版不补前后月灰色日期，取舍的是“完整月历观感”与“侧边栏稳定布局”之间的平衡；当前更优先后者
- 组件做成可挂载结构会比一次性内联脚本稍多一点前期组织成本，但长期维护收益更高
- `lunar-javascript` 能满足当前需求，但若未来扩展到更复杂的节假日排班或多日历体系，可能需要额外抽象

## 结论

本次设计的最终方向是：

- 使用 `lunar-javascript` 作为黄历数据能力来源
- 在 `argon` 主题中新增一个可复用的日历组件
- 将组件拆分为独立模板、脚本、样式三层
- 首先在侧边栏卡片中接入

这样可以在尽量少破坏现有主题结构的前提下，做出一个既符合用户当前需求、又方便后续被其他模块直接引入的原生日历组件。
