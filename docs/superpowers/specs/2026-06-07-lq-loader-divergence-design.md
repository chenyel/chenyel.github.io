# 科幻仪表加载页组件设计

## 背景

当前博客使用 `Hexo + argon` 主题，已经有一套轻量的加载页实现，入口位于 `themes/argon/layout/_partial/lq-effects.ejs`，并通过 `source/_data/argon.yml` 中的 `lq_loader` 配置控制。

用户希望在 `D:/blog` 现有主题基础上，将加载页升级为一套更有世界观感的科幻仪表风方案，同时满足以下目标：

- 只改加载页，不改首页首屏和正文结构
- 默认全站启用
- 风格为“二创版”，保留科幻仪表气质，但不直接照搬原作表现
- 主视觉采用冷青蓝配色
- 文案使用中英混合表达
- 数字区采用随机仪表值，而不是真实时间
- 页面加载完成后先显示一个短暂完成态，再退场
- 在实现上尽量隔离、组件化，方便以后复用和扩展
- 默认走全局配置，必要时允许页面 front-matter 覆盖全局

## 目标

- 为 `argon` 主题设计一套独立的科幻仪表加载页组件。
- 在不重构整套主题结构的前提下，提升加载页的视觉完成度与个性表达。
- 保持组件边界清晰，避免把结构、样式、脚本和配置继续揉在现有效果文件中。
- 为后续同类加载页扩展预留接口，而不是一次性写死。

## 非目标

- 第一版不改首页 hero、侧边栏、文章卡片或正文排版。
- 第一版不做多套加载页模板切换 UI。
- 第一版不引入外部动画库或第三方依赖。
- 第一版不让页面覆盖修改组件结构或脚本流程，只允许覆盖文案与风格参数。

## 现状摘要

- 当前加载页模板和其他视觉效果一起放在 `themes/argon/layout/_partial/lq-effects.ejs`。
- 当前加载页是一张简单文案卡片，不具备独立的数字仪表视觉。
- 主题资源目前主要集中在 `themes/argon/source/style.css`、`themes/argon/source/argontheme.js` 和若干 `lq-*` 文件中。
- 当前项目目录下不存在 `D:/blog/.git`，仅存在部署产物目录，因此本次设计文档可以落盘，但无法按常规流程提交 commit。

## 设计总览

本次加载页采用“独立组件 + 薄装配层”的方式接入：

1. 独立 partial 负责输出加载页 HTML 结构
2. 独立 CSS 负责仪表风视觉与退场动画
3. 独立 JS 负责数字跳动、完成态和隐藏逻辑
4. `lq-effects.ejs` 只负责读取配置、挂载组件和下发运行时配置

这样可以把“是否启用加载页”和“加载页内部如何实现”解耦，避免后续继续把逻辑堆进统一 effects 模板。

## 组件边界

### 模板层

新增独立 partial，例如：

- `themes/argon/layout/_partial/loaders/lq-divergence-loader.ejs`

该模板只负责：

- 输出加载页根节点
- 输出标题、副标题、状态文案容器
- 输出数字仪表框架
- 输出用于脚本读取的配置数据属性或 JSON

模板层不承载复杂状态逻辑，不直接写随机数字算法。

### 样式层

新增独立样式文件，例如：

- `themes/argon/source/css/lq-divergence-loader.css`

样式全部使用独立前缀，例如：

- `.lq-divergence-loader-*`

这样可以避免污染现有 `argon` 样式，也便于以后单独迁移、替换或禁用。

### 脚本层

新增独立脚本文件，例如：

- `themes/argon/source/js/lq-divergence-loader.js`

脚本只负责以下职责：

- 初始化加载页
- 驱动随机数字跳动
- 处理最短展示时长
- 监听页面加载完成
- 进入完成态并切换文案
- 退场并释放页面滚动

脚本不依赖外部库，采用原生 JavaScript 实现。

### 装配层

保留 `themes/argon/layout/_partial/lq-effects.ejs` 作为统一注入点，但只承担薄装配职责：

- 判断 `show_lq_loader`
- 读取最终配置
- include 对应 loader partial
- 输出组件运行时配置

这样后续如果新增别的加载页风格，可以继续复用同一入口，而不是重新找挂载点。

## 配置设计

本次在保留现有 `show_lq_loader` 和 `lq_loader` 命名习惯的前提下，升级配置结构。

建议字段包括：

- `show_lq_loader`
  - 全局开关
- `lq_loader.mode`
  - 加载页类型，第一版固定为 `divergence`
- `lq_loader.scope`
  - 默认启用范围，第一版设为 `site`
- `lq_loader.minimum_duration`
  - 首次访问最短展示时长
- `lq_loader.repeat_visit_duration`
  - 重复访问时较短展示时长
- `lq_loader.brand`
  - 品牌或站点标识文案
- `lq_loader.title`
  - 主标题，偏世界观表达
- `lq_loader.subtitle`
  - 默认副标题或加载中文案
- `lq_loader.complete_text`
  - 完成态提示文案
- `lq_loader.palette`
  - 颜色组，例如主发光色、背景色、边线色
- `lq_loader.meter`
  - 仪表配置，例如位数、刷新频率、主位前缀、随机策略
- `lq_loader.overrides`
  - 路径级覆盖配置表

## 配置优先级

最终运行时配置采用单向合并，优先级如下：

`front-matter > lq_loader.overrides > lq_loader 全局默认`

这意味着：

- 平时只维护全局默认配置即可
- 需要按页面路径定制时，优先在 `argon.yml` 的 `overrides` 中处理
- 个别页面需要进一步特殊化时，再使用 front-matter 局部覆盖

页面覆盖仅允许修改文案、颜色、仪表参数等配置值，不允许修改模板结构和状态机流程。

## 视觉设计

### 整体气质

整体方向为“个人世界入口”的科幻仪表风，而不是直接复刻影视或动画中的经典仪表盘。

最终视觉应具备以下特征：

- 深色全屏背景
- 冷青蓝主发光色
- 轻量的网格、扫描线和边缘微光
- 横向数字仪表区作为主视觉
- 中英混合文案营造系统界面感

### 背景层

背景采用多层轻量叠加：

- 深色基底
- 冷青蓝径向微光
- 细网格或重复线性纹理
- 低强度扫描覆盖层

背景不引入插画、角色图或大体积图片，以免压过正文站点的主视觉体系。

### 仪表区

中央仪表区为横向面板，包含：

- 标题区域
- 数字仪表主区
- 状态文案区

数字主区采用多个等宽格组成，主位与小数位可在视觉上做层级区分，但数字内容仅作为随机仪表值使用，不对应真实时间。

### 文案风格

文案采用中英混合，并强调世界观感。

建议语气特征：

- 不是传统“页面加载中”
- 更接近“同步中”“进入中”“校准中”
- 英文更像系统标签，中文更像叙事提示

主标题、副标题和完成态文案应彼此统一，不使用强娱乐化或过度中二的表达。

## 交互设计

### 初始进入

- 页面开始渲染后立即显示加载层
- 锁定页面滚动
- 数字区进入高速跳动状态
- 状态文案显示默认加载提示

### 运行中

- 数字区按固定节奏刷新，节奏稳定，不做过于夸张的跳闪
- 随机值只负责氛围，不与页面真实数据绑定
- 页面主体在底层继续完成正常加载

### 完成态

当页面加载完成且达到最短展示时长后，不立即消失，而是先进入 `complete` 状态：

- 数字跳动减速或短暂定格
- 状态文案切换为完成提示
- 可追加轻微亮度提升或扫描收束动画

完成态持续时间应短促明确，作为一次“仪表完成校准”的视觉提示，而不是额外停留太久。

### 退场

完成态结束后：

- 加载层整体淡出
- 恢复页面滚动
- 在过渡结束后移除或隐藏组件节点

退场必须顺畅，不应造成明显卡顿或阻塞首屏操作。

## 状态机设计

脚本层建议采用明确状态机，而不是多个零散定时器拼接。

建议状态包括：

- `idle`
  - 组件已挂载但尚未开始
- `running`
  - 数字跳动中，等待页面加载完成
- `complete`
  - 加载完成，进入短暂完成演出
- `hidden`
  - 已退场，不再占据交互层

采用显式状态机的收益：

- 完成态逻辑更稳定
- 以后扩展新模式时不容易出现计时器竞态
- 调试和复用更简单

## 启用范围

第一版默认全站启用，即：

- 首页启用
- 独立页面启用
- 文章页启用
- 归档、标签、分类等页面也启用

后续若用户希望收窄范围，可通过 `scope` 和覆盖配置控制，而不是改组件代码。

## 复用与扩展约束

为了保证这次实现可长期维护，需遵守以下约束：

- 模板、样式、脚本分离，不把逻辑重新塞回 `style.css` 与 `argontheme.js`
- 组件类名必须独立前缀化
- `lq-effects.ejs` 只做装配，不做复杂行为实现
- 配置只做单向合并，不在脚本里判断“配置来自哪里”
- 页面级覆盖只改参数，不改结构
- 第一版即预留 `mode` 字段，为未来新增其他加载页风格留出口

## 文件落位建议

建议改动点如下：

- `themes/argon/layout/_partial/loaders/lq-divergence-loader.ejs`
  - 加载页模板骨架
- `themes/argon/layout/_partial/lq-effects.ejs`
  - 接入新组件并输出运行时配置
- `themes/argon/source/css/lq-divergence-loader.css`
  - 组件独立样式
- `themes/argon/source/js/lq-divergence-loader.js`
  - 组件独立脚本
- `source/_data/argon.yml`
  - 全局默认配置、路径覆盖配置

若需要页面级 front-matter 覆盖，建议在实现阶段统一约定字段命名，避免出现与其他页面参数混淆的简写。

## 错误处理与边界

- 若配置缺失，组件应回退到安全默认值，而不是整块不渲染
- 若脚本初始化失败，页面主体仍需可正常访问
- 若某些页面设置关闭加载页，不应影响 cursor、snow 等其他效果初始化
- 若用户开启页面级覆盖但字段不完整，应只覆盖已提供字段，其余回退全局默认
- 若浏览器性能较弱，优先保证退场顺畅，避免堆叠过重动画

## 验证标准

完成实现后应满足以下标准：

1. 加载页默认在全站出现，不破坏页面主体结构
2. 视觉风格明显升级为冷青蓝科幻仪表风，但不直接照搬外部作品现成界面
3. 中央数字区显示随机仪表值，而非真实时间
4. 文案采用中英混合表达，整体气质统一
5. 页面加载完成后先进入短暂完成态，再顺畅退场
6. 组件结构、样式、脚本彼此独立，可在后续被其他页面或风格复用
7. 全局配置生效，且页面 front-matter 能覆盖全局参数
8. 主题其他效果逻辑不因本次改动被耦合或破坏

## 风险与取舍

- 采用独立组件方案会比直接就地改现有 loader 多一些组织成本，但长期可维护性明显更好
- 第一版默认全站启用，统一性更强，但也意味着文章页访问会频繁看到加载层；这是当前用户有意选择的体验方向
- 不引入外部依赖能降低复杂度，但也意味着某些复杂动画效果需自行控制，视觉表现要在精简和完成度之间平衡
- 允许 front-matter 覆盖会增加配置分支，但通过限制覆盖范围为“参数级覆盖”，可以把复杂度控制在可维护范围内

## 结论

本次设计的最终方向是：

- 在 `argon` 主题中新增一套独立的科幻仪表加载页组件
- 默认全站启用，保留现有 `show_lq_loader` / `lq_loader` 配置入口
- 使用冷青蓝、中英混合、随机仪表值和完成后退场的交互节奏
- 将实现拆分为独立 partial、独立样式、独立脚本和统一装配入口
- 采用 `front-matter > overrides > 全局默认` 的配置优先级

这样可以在尽量少破坏现有主题结构的前提下，把当前加载页升级为一套更有辨识度、也更方便后续复用的主题级组件。
